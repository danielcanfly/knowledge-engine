## Objective

Execute one bounded credential-bearing, read-only live acceptance against the reconciled candidate collection. The run verifies that the same frozen twenty-four R3.5 query variants produce the accepted quality outcome when their embeddings are generated by live Cloudflare Workers AI and their dense candidates are read from live Qdrant.

The live result remains non-production evidence. It cannot serve answers, influence lexical output, mutate the candidate, or clear a blocker without later evidence sealing and independent reconciliation.

## Collection identity and immutability

Before any live query, the operator verifies the candidate collection schema and performs complete readback of all `107` expected point IDs with payloads and named vectors. It compares the exact ID set and float32 point fingerprints with the reconciled R3.6 manifest.

After all twenty-four live queries, it repeats the collection snapshot and complete `107`-point readback. The pre-query and post-query schema, ID set, payloads, vectors and aggregate fingerprint must be identical.

The historical pilot collection `llm_wiki_m23_pilot_bge_m3_1024` is outside the operation. The client exposes only Qdrant `GET` and `POST` read surfaces. It has no create, update, upsert, delete or reindex operation.

## Live query execution

The frozen eight probes each contain three distinct query variants, for twenty-four unique query identities. Every variant is executed separately:

1. send one query to Cloudflare Workers AI model `@cf/baai/bge-m3`;
2. validate the returned `1024`-dimensional L2-normalized embedding;
3. query the candidate collection through named vector `default` for the top `50` dense candidates;
4. validate every returned payload as payload schema v2, candidate-only, noncanonical and nonproduction;
5. record provider, Qdrant and total latency for that individual variant.

Exactly one bounded Cloudflare HTTP session and one bounded Qdrant HTTP session are reused for the run and explicitly closed. Batch embedding is not used, so the latency budget cannot be hidden by request amortisation.

## Accepted ranking path

The live Qdrant results supply dense rankings only. For each three-variant probe, the operator applies the accepted R3.5 target-unaware ranking path locally:

- positive-match BM25 over query-visible semantic fields;
- reciprocal-rank fusion of the three dense rankings;
- dense top-ten consensus credit;
- query-class lexical weighting;
- deterministic score and section-ID ordering.

The ranker does not accept target section IDs, expected relevance labels, offline case IDs or probe IDs as scoring inputs.

## Frozen quality gates

The live result must satisfy all of the following without changing thresholds:

- Recall@5 at least `0.82`;
- MRR@10 at least `0.68`;
- nDCG@10 at least `0.72`;
- exact metric parity with accepted R3.5 values `0.875`, `0.807291666667`, and `0.851933109598`;
- exact target-rank parity with the eight accepted R3.5 cases;
- maximum top-ten hub frequency no greater than `6`;
- twenty-four unique query identities;
- zero query errors and zero ACL violations;
- exact pre-query and post-query `107`-point identity parity.

## Latency gate

The existing canonical bounded-live budget is retained unchanged. End-to-end latency p95 across the twenty-four individually executed live variants must be no greater than `1200 ms`.

A quality-passing run that exceeds the latency gate is recorded as `completed_fail_closed_live_acceptance`. The result remains valid evidence and carries `blocked_pending_latency` in addition to the retrieval-quality blocker.

## Privacy contract

Durable evidence may contain query digests, variant IDs, ranked section IDs, aggregate metrics, bounded latency values, collection identities and failure classes.

It must not contain raw query text, raw answers, document text, credentials, service URLs, hostnames or arbitrary exception messages.

## Authority boundary

This stage authorizes Cloudflare query embeddings and Qdrant read-only operations against the exact candidate collection. It authorizes no Qdrant write, delete or reindexing, no historical-pilot or production collection mutation, and no R2, pointer, Source or production mutation.

Production retrieval remains lexical. Serving, traffic, deployment, threshold changes, promotion eligibility, retrieval-quality blocker clearance, parent R3 closure and M23.7 closure remain false.

## Completion rule

A real operator receipt is required after the implementation PR merges. Both `pass_live_acceptance` and `completed_fail_closed_live_acceptance` are sealable outcomes. The implementation issue may close only after the real receipt is sealed, exact-head CI passes, and an independent reconciliation merges.
